# 担当連絡（顧客↔担当エージェント リアルタイムチャット）詳細設計書

- 版: v1.0（2026-06-18）
- 対象: AI不動産名刺（ai-fcard.com / 素のPHP + PDO/MySQL）
- 位置づけ: AI担当機能の次に整備する優先機能
- 確定方針:
  - **顧客の本人識別 = SMS電話認証を必須化**（既存 Firebase SMS 認証基盤を流用）
  - **リアルタイム配信 = ポーリング方式**（共有ホスティング前提の現実解）
  - 着手順: 本設計書 → Phase 1 実装

---

## 0. 用語と前提

| 用語 | 意味 |
|---|---|
| 顧客 | 名刺ページ（card.php）の訪問者。アカウントは持たず、**SMS認証した電話番号で同定**する |
| 担当 / エージェント | 名刺の保有者。`users` でログイン済み（`$_SESSION['user_id']`）。マイページ（edit.php）からアクセス |
| ボット発言 | 既存AIチャットの自動応答（`chat_messages.role='bot'`） |
| 担当発言 | **本機能で新設する人間の担当からの発言**（`role='agent'`） |
| セッション | `chat_sessions`（1名刺×1顧客に1本、SMS認証で別端末からも復帰） |

### 最重要の前提（設計の土台）

現在の card.php のチャットは **「顧客 ↔ AIボット」** の都度リクエスト応答であり、リアルタイム機構は存在しない。
本機能の担当連絡は **「顧客 ↔ 人間の担当」** であり、**同じ `chat_sessions` / `chat_messages` を共有チャネルとして使い、`role` で発言主体を区別する**方針とする（別テーブルにしない）。これにより、AI記憶連携・顧客一覧・履歴表示の既存資産をそのまま活かせる。

## 2. データモデル変更

### 2.1 `chat_messages` の拡張（マイグレーション）

```sql
-- migrations/20260618_add_agent_messaging.sql

-- 1) 役割に 'agent'（人間の担当発言）を追加
ALTER TABLE chat_messages
  MODIFY COLUMN role ENUM('user','bot','system','agent') NOT NULL;

-- 2) 送信者（担当発言時の users.id。user/bot/system は NULL）
ALTER TABLE chat_messages
  ADD COLUMN sender_user_id INT NULL DEFAULT NULL AFTER role;

-- 3) 既読管理（顧客が読んだ時刻 / 担当が読んだ時刻）
--    顧客発言→担当が読む / 担当発言→顧客が読む の双方向を1カラムで扱う
ALTER TABLE chat_messages
  ADD COLUMN read_at TIMESTAMP NULL DEFAULT NULL AFTER message;

-- 4) 添付（メッセージ:添付 = 1:N。本文なし添付のみ送信も許可）
CREATE TABLE IF NOT EXISTS chat_message_attachments (
  id INT AUTO_INCREMENT PRIMARY KEY,
  message_id INT NOT NULL,
  session_id CHAR(36) NOT NULL,
  kind ENUM('image','pdf','word','excel','other') NOT NULL,
  original_name VARCHAR(255) NOT NULL,
  stored_path VARCHAR(512) NOT NULL,     -- backend/uploads/chat/ 配下の相対パス
  mime_type VARCHAR(127) NOT NULL,
  byte_size INT NOT NULL,
  width INT NULL,                        -- 画像のみ
  height INT NULL,                       -- 画像のみ
  thumb_path VARCHAR(512) NULL,          -- 画像のみ（一覧表示用）
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (message_id) REFERENCES chat_messages(id) ON DELETE CASCADE,
  FOREIGN KEY (session_id) REFERENCES chat_sessions(id) ON DELETE CASCADE,
  INDEX idx_attach_message (message_id),
  INDEX idx_attach_session (session_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- 5) 未読集計を速くするための複合インデックス
ALTER TABLE chat_messages
  ADD INDEX idx_unread (session_id, role, read_at);
```

#### 既読モデルの定義（重要）
- `read_at` は「**その発言を相手が読んだ時刻**」。
  - 顧客発言（`role='user'`）の `read_at` = 担当が読んだ時刻。
  - 担当発言（`role='agent'`）の `read_at` = 顧客が読んだ時刻。
  - `bot` / `system` は対象外（常に NULL）。
- 顧客側の未読件数 = `role='agent' AND read_at IS NULL` の件数。
- 担当側の未読件数 = `role='user' AND read_at IS NULL` の件数。

### 2.2 既存テーブルへの影響
- `chat_crm_cases` / `chat_session_memory` は **変更不要**。担当発言も `send.php`/新エンドポイント側で `chatCrmSyncFromChatSession` を呼び、AI要約へ反映する。
- 既存のボット応答ロジックは `role` を `'user'/'bot'` でしか書き込まないため、enum拡張は後方互換。

## 3. API設計

ベース: `backend/api/chat/`。担当側エンドポイントは `requireAuth()` + **セッション所有者検証**（`bc.user_id = $userId`）必須。顧客側は session_id + visitor_id（既存の公開エンドポイント流儀）。

### 3.1 担当側（マイページ）

| メソッド/パス | 用途 | 認証 | 主な入出力 |
|---|---|---|---|
| `GET sessions.php` | 顧客一覧（**未読件数・要返信フラグ追加**） | 要 | 既存に `unread_count`, `last_message_role`, `last_message_at`, `needs_reply` を追加。`needs_reply DESC, last_message_at DESC` で並べ替え |
| `GET session.php` | 会話全表示（**agent発言・添付を含める**） | 要 | 既存に attachments を付与 |
| **`POST agent/send.php`（新規）** | 担当→顧客の送信 | 要 | in: `{session_id, message?, attachment_ids?[]}` out: `{message_id, created_at}`。送信時に当該セッションの顧客発言を既読化（`role='user'` の `read_at` を now に）+ `chatCrmSyncFromChatSession` 実行 |
| **`POST agent/read.php`（新規）** | 顧客発言を既読化 | 要 | in: `{session_id}`。`role='user' AND read_at IS NULL` を一括既読 |
| **`GET agent/poll.php`（新規）** | 担当の新着ポーリング | 要 | in: `?since_id=` （任意 `?session_id=`）out: 当該担当の全セッション横断の新着 `user` 発言と未読合計。`since_id` で差分取得 |
| `POST reply-draft.php` | 自動回答作成 / ブラッシュアップ | （**要に変更**） | 既存。`mode=auto`/`polish`。**現状は無認証なので所有者検証を追加**（セキュリティ修正） |
| **`POST attachment/upload.php`（新規）** | 添付アップロード（担当・顧客共用） | 要 or 顧客検証 | multipart。検疫→リサイズ→`chat_message_attachments` 仮登録し `attachment_id` 返却（後続 send で確定） |

### 3.2 顧客側（card.php ウィジェット）

| メソッド/パス | 用途 | 認証 | 主な入出力 |
|---|---|---|---|
| `POST session/start.php` | セッション開始/復帰 | SMS必須 | 既存。**未認証ならボット応答前に SMS 認証を促す**（後述4.1） |
| `POST phone/verify.php` | SMS認証→電話番号で復帰 | — | 既存流用 |
| `POST send.php` | 顧客→（ボット/担当）送信 | session+visitor | 既存。本機能では担当宛メッセージも同じく `role='user'` で保存。**ハンドオフ状態（4.3）に応じてボット応答を抑制** |
| **`GET customer/poll.php`（新規）** | 顧客の新着ポーリング | session+visitor | in: `?session_id=&visitor_id=&since_id=` out: 新着 `agent` 発言（添付込み）+ 未読件数。取得時に担当発言を既読化 |
| `POST attachment/upload.php` | 添付アップロード（共用） | session+visitor | 上と同じ |

### 3.3 ポーリング仕様（共通）
- **間隔**: 画面アクティブ時 4 秒 / バックグラウンド（`document.hidden`）時 20 秒。`since_id`（最後に受信した `chat_messages.id`）で差分のみ取得。
- **負荷対策**: `poll.php` は SELECT のみで OpenAI 等の重処理を呼ばない。`idx_unread` を使った件数クエリのみ。
- **将来差し替え**: 取得I/Fを `since_id` 差分で固定しておくことで、後日 SSE/WebSocket に置換しても画面側の契約を変えずに済む。

## 4. 画面フロー

### 4.1 顧客側フロー（card.php / chat-widget.js）
```
名刺ページ訪問
  ↓
チャット起動 →（SMS未認証なら）SMS認証必須（既存 phone/verify.php, reason='upfront'）
  ↓ 認証成功（電話番号でセッション復帰 or 新規）
LINE風チャット画面
  ├─ 吹き出し（左=担当/ボット, 右=顧客）
  ├─ 送信日時表示
  ├─ 既読/未読表示（担当発言を顧客が見た時点で担当側に既読反映）
  ├─ 添付送信（画像/PDF/Word/Excel、画像は自動リサイズ）
  └─ ポーリングで担当の新着を受信（タブ通知/ブラウザ通知）
```
- **担当連絡メニューの未読バッジ**: `customer/poll.php` の `unread_count`（`role='agent'` 未読）を `担当連絡（3）` のように表示。難しければ「担当からの新着があれば自動で最前面表示」のフォールバック（要件1-2の代替案）も可。

### 4.2 担当側フロー（edit.php → チャット履歴）
```
マイページ（edit.php）
  ↓
チャット履歴（顧客一覧 = sessions.php）
  ├─ 各行に未読バッジ（例: 山田太郎 ●3）
  ├─ 要返信（顧客発言が未読）を上位表示
  ↓ 顧客選択
担当連絡画面（session.php で全履歴）
  ├─ メッセージ/添付/送信者/日時/既読状況
  ├─ 【自動回答作成】ボタン → reply-draft.php(mode=auto) → 編集欄へ
  ├─ 【ブラッシュアップ】ボタン → reply-draft.php(mode=polish) → 編集欄へ
  └─ 【送信】→ agent/send.php
```

### 4.3 ボット↔担当のハンドオフ（重要設計）
1セッション内にボット応答と担当応答が混在するため、状態を持たせる。`chat_crm_cases` に軽量フラグを追加（JSON `progress` 内 or 新カラム `handoff_mode ENUM('bot','agent') DEFAULT 'bot'`）。

- `handoff_mode='bot'`（既定）: 従来どおり `send.php` がボット応答。
- `handoff_mode='agent'`: 担当が会話に参加中。`send.php` は**ボット応答を生成せず**顧客発言を保存し未読として担当に通知（誤って二重応答しない）。
- 切替: 担当が `agent/send.php` で初回送信した時点で自動的に `agent` へ。担当画面に「AI応答に戻す」トグルを用意。

> これにより「LINEのような担当との往復」と「AIボット相談」を1画面で両立し、要件6（AI自動返信・予約など将来拡張）にも段階移行できる。

## 5. 添付ファイル仕様（要件1-3）

| 種別 | 拡張子/MIME | 処理 |
|---|---|---|
| 画像 | jpeg/png/webp/gif | **長辺2,000px以内へ自動リサイズ + 圧縮**（GD/Imagick、品質85%）。一覧用サムネ生成 |
| PDF | application/pdf | 検疫後そのまま保存。画面はDLリンク（初期はプレビューなし） |
| Word | .doc/.docx | 同上 |
| Excel | .xls/.xlsx | 同上 |

- 保存先: `backend/uploads/chat/{business_card_id}/{session_id}/`。`.htaccess` で直アクセス制限し、配信は認証付き `attachment/download.php` 経由（顧客は session+visitor 検証、担当は所有者検証）。
- 検疫・MIME検証・サイズ上限・レート制限は [upload_security.php](../backend/includes/upload_security.php) を流用（上限10MB踏襲）。
- 画像リサイズは既存のプロフィール写真処理と同じ GD/Imagick + EXIF 自動回転ロジックを共通関数化して再利用。

## 6. AI機能の接続（要件3・4・5）

### 6.1 自動回答作成 / ブラッシュアップ
- 既存 [reply-draft.php](../backend/api/chat/reply-draft.php) を担当連絡画面から呼ぶだけ。
  - 【自動回答作成】= `mode=auto`（最新顧客メッセージ+履歴+CRM要約から文案生成 → 編集欄へ）
  - 【ブラッシュアップ】= `mode=polish`（担当入力文を改善 → 編集欄へ）
- **修正点（セキュリティ）**: 現状 reply-draft.php は認証なしで session_id だけで呼べる。`requireAuth()` + セッション所有者検証を追加する。

### 6.2 AI記憶連携（顧客プロフィール要約）
- 担当連絡の発言（user/agent 双方）も、保存時に `chatCrmSyncFromChatSession()` を呼んで `chat_crm_cases.ai_summary` / `conditions_json` を更新。
- 保存対象（要件5-1: 希望条件・家族構成・購入/売却理由・希望時期・資金計画・ローン状況・検討エリア・NG条件・趣味嗜好等）は、既存CRM要約の抽出プロンプトに担当連絡の文脈も含めることで充足。
- 通常AIチャット（card.php）応答時、既存どおり `chat_crm_cases` / `chat_session_memory` を参照するため、「前回話した予算で探して」が成立。**追加実装は最小**。

## 7. 通知（要件1-4）

- **Phase 1**: タブが開いている間の通知。ポーリングで新着検知時に
  - 担当側: 画面内トースト + 顧客一覧バッジ更新。

- **Phase 4（将来）**: 閉じていても届く Web Push（VAPID + Service Worker [sw.js](../sw.js) + 購読管理テーブル）。中程度の重さのため後回し。

## 8. セキュリティ / 権限

- 担当側エンドポイント（agent/send, agent/read, agent/poll, session, sessions, reply-draft）は `requireAuth()` + `business_cards.user_id = $userId` 検証を必須。
- 顧客側は session_id 形式検証（既存の36桁UUID正規表現）+ visitor_id 検証 + SMS認証済み電話の突合。
- 添付は MIME ホワイトリスト + 拡張子 + サイズ + 画像ピクセル爆弾対策（既存流用）、保存ディレクトリは実行権限なし（`.htaccess`）。
- 配信は必ず認証付きプロキシ経由。直リンク禁止。
- ポーリングI/Fはレート制限（IP+セッション単位）。
