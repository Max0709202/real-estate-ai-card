# 本番データ保全FAQ

更新日: 2026-07-09

## 顧客向け回答案

### 1. データベースは自動バックアップされていますか。取得頻度と保存期間を教えてください。

はい。データベースは毎日1回、自動バックアップを取得する運用にします。保存期間は35日間を標準とし、必要に応じて月次バックアップを別途長期保管します。

### 2. バックアップから実際に復元できることは確認済みでしょうか。

バックアップ取得後、定期的に検証用データベースへ復元テストを行い、SQLファイルが破損していないこと、主要画面が参照できることを確認します。復元テストは初回設定時と、その後は少なくとも月1回実施します。

### 3. システム更新前には、データベースとソースコードのバックアップを取得していただいていますか。

はい。システム更新前には、データベースのダンプとソースコードのアーカイブを取得してから作業します。更新に問題が発生した場合は、更新前の状態へ戻せるようにします。

### 4. ソースコードはGit等でバージョン管理され、必要に応じて以前の状態へ戻せるようになっていますか。

はい。ソースコードはGitで管理し、変更履歴を残します。必要に応じて過去のコミットへ戻せる運用にします。

### 5. 本番環境を直接修正する運用ではなく、開発環境・テスト環境を経由してリリースする運用になっていますか。

原則として、開発環境で修正し、動作確認後に本番環境へ反映します。本番環境での直接修正は緊急時に限定し、実施した場合もGitへ反映して履歴を残します。

### 6. 万一障害が発生した場合、どの程度の時間で復旧できる想定でしょうか。

障害内容によりますが、アプリケーション更新に起因する不具合は30分から2時間程度で切り戻しを目指します。データベース復元が必要な障害は、バックアップサイズやサーバー状況により2時間から半日程度を目安とします。

### 7. サーバー障害やランサムウェア等に対する対策はどのようになっていますか。

データベースとソースコードのバックアップを本番公開ディレクトリ外に保存し、ハッシュ値で破損確認できるようにします。加えて、定期的に外部ストレージ等へ二重保管することで、サーバー障害やランサムウェア被害時の復旧性を高めます。管理画面・サーバー管理画面のパスワード管理、不要アカウントの削除、PHP/ライブラリ更新も継続します。

### 8. 操作ログや監査ログは保存されていますか。

管理者による変更履歴、アクセスログ、メール送信ログ、チャット関連ログなどを保存する設計があります。今後は重要操作について「誰が、いつ、何を変更したか」を追跡できるよう、監査ログ対象を継続的に拡張します。

## 現状確認メモ

- Gitリポジトリは存在し、`main` と `origin/main` が紐づいています。
- cronにはチャット知識同期と通知キュー処理が登録されています。
- DBバックアップcronは未登録だったため、以下の手順で追加します。
- `backend/config/config.php` は `ENVIRONMENT` が `development` のままです。本番では `production` へ変更してください。
- `session.cookie_secure` が `0` です。HTTPS本番では `1` へ変更してください。

## 初心者向け実装手順

### Step 1. バックアップを手動で1回実行する

```bash
cd /home/xs013436/ai-fcard.com/public_html
/usr/bin/php backend/scripts/backup_database.php
/usr/bin/php backend/scripts/backup_source.php
```

### Step 2. バックアップファイルを確認する

```bash
ls -lh /home/xs013436/ai-fcard.com/backups/db
ls -lh /home/xs013436/ai-fcard.com/backups/source
```

`.sql.gz` がデータベース、`.tar.gz` がソースコードです。`.sha256` は破損確認用のハッシュです。

### Step 3. cronに自動バックアップを追加する

```bash
crontab -e
```

以下を追加します。

```cron
5 2 * * * cd /home/xs013436/ai-fcard.com/public_html && /usr/bin/php backend/scripts/backup_database.php --keep-days=35 >> /home/xs013436/ai-fcard.com/backups/backup.log 2>&1
20 2 * * * cd /home/xs013436/ai-fcard.com/public_html && /usr/bin/php backend/scripts/backup_source.php --keep-days=35 >> /home/xs013436/ai-fcard.com/backups/backup.log 2>&1
```

### Step 4. 復元テスト用DBを作る

本番DBに直接戻す前に、必ずテスト用DBへ復元します。サーバー管理画面で `restore_test` のようなテスト用DBを作成してください。

### Step 5. 復元テストを実行する

```bash
cd /home/xs013436/ai-fcard.com
gunzip -c backups/db/バックアップファイル名.sql.gz | mysql -h localhost -u ユーザー名 -p テスト用DB名
```

復元後、主要テーブルの件数を確認します。

```bash
mysql -h localhost -u ユーザー名 -p テスト用DB名 -e "SHOW TABLES;"
```

### Step 6. システム更新前の作業ルール

```bash
cd /home/xs013436/ai-fcard.com/public_html
/usr/bin/php backend/scripts/backup_database.php
/usr/bin/php backend/scripts/backup_source.php
git status --short
```

更新前にこの3つを必ず実行します。`git status` で未コミット変更がある場合は、内容を確認してから更新します。

### Step 7. 外部保管を追加する

サーバー内バックアップだけでは、サーバー全体障害やランサムウェアに弱いため、別サーバー・Google Drive・S3等へ定期コピーします。最低でも週1回、できれば毎日コピーしてください。

### Step 8. 月1回の復元訓練を予定に入れる

毎月1回、最新バックアップをテストDBへ復元し、復元できた日時・担当者・結果を記録します。
