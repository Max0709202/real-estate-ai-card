## 0. 先方ご指摘への総括

関連会社様からのご指摘は、技術的に妥当な観点を含んでいます。特に、AI API費用、RAG/Web参照、機微情報、暗号化、会話品質の継続保守、共有サーバーでの運用限界は、リリース前に明文化しておくべき論点です。

一方で、現行コードベースを確認すると、以下の通り「毎回、無制限に膨大な全文コンテキストをLLMへ送る設計」ではありません。

- 会話履歴は `backend/api/chat/send.php` で直近20メッセージに制限しています。
- AI回答の出力上限は `backend/includes/openai-chat-helper.php` で `max_tokens = 1024` に制限しています。
- RAG参照は `getChatRagContextForChat($db, $message, 6)` により最大6チャンクを選択します。
- 1チャンクあたり本文抜粋は最大900文字、RAG全体は約4,800文字で打ち切ります。
- 最新情報が必要な質問のみ、最大4件の公式ソースを24時間単位で再取得します。
- 購入/売却/ローン等のヒアリングは、まずPHP側の構造化ロジックで処理し、常にLLMへ投げるわけではありません。
- OpenAIの組み込みWeb検索ツールは現行コードでは使用していません。国税庁・国交省等の公式ページをサーバー側で取得し、MySQLにキャッシュした抜粋をRAGとして渡しています。

したがって、回答方針としては次の通りです。

1. MVP/小規模運用は、現行のXserver上でも技術的には運用可能です。
2. ただし、年収、自己資金、ローン残債、告知事項等を本格的に保存・活用する正式運用では、暗号化、監査ログ、アクセス制御、脆弱性診断、保存期間管理が必要です。
3. Xserver継続運用の場合は、アプリケーションレイヤー暗号化と運用ルールで補完します。
4. 大量流入、厳格な監査、KMS管理、WAF詳細制御、DB暗号化証跡が必要になった段階で、AWS等の専用環境へ移行します。

## 2. 1）コストシミュレーション

### 2-1. 結論

APIトークン費用そのものは、現行モデル `gpt-4o-mini` 前提では比較的低額です。費用の中心は、API代ではなく、以下です。

- 会話品質レビュー
- RAGソース更新
- セキュリティ運用
- 障害監視
- モデル変更時の回帰テスト
- 個人情報管理

### 2-2. OpenAI API単価

2026年5月23日時点で確認したOpenAI公式情報を前提にします。

現行コードの標準モデル `gpt-4o-mini`:

- 入力: 0.15 USD / 100万 tokens
- 出力: 0.60 USD / 100万 tokens

将来、品質向上目的で `gpt-5.4-mini` クラスへ変更する場合の参考:

- 入力: 0.75 USD / 100万 tokens
- 出力: 4.50 USD / 100万 tokens

為替は保守的に `1 USD = 155円` で試算します。

参考:

- OpenAI API Pricing: https://developers.openai.com/api/docs/pricing
- OpenAI GPT-4o mini release/pricing: https://openai.com/index/gpt-4o-mini-advancing-cost-efficient-intelligence/

### 2-3. 1AI応答あたりのトークン想定

現行コードの制限値から、1AI応答あたりのトークンを以下のように見ます。

| パターン | 入力tokens | 出力tokens | 内容 |

| ヒアリングのみ | 0 | 0 | PHP側で定型質問/クイック返信を返す |
| 軽い一般相談 | 3,000 | 400 | 履歴少なめ、RAGなし |

本資料では、標準ケースを `入力5,500 tokens / 出力650 tokens` として月額試算します。

### 2-6. Web検索/RAGの費用

現行コードは、OpenAIの組み込みWeb検索ツールを使っていません。そのため、OpenAI Web Search tool の検索課金は発生しません。

現行方式:

- 公式URLを当社サーバーから取得
- MySQLの `chat_knowledge_sources` / `chat_knowledge_chunks` に保存
- 会話時は関連チャンクだけをLLMへ渡す
- 追加費用は主にサーバーCPU/DB/cron/保守工数

もし将来OpenAIのWeb検索ツールを使う場合:

- OpenAI Web search: 10 USD / 1,000 calls
- 1 callあたり: 0.01 USD = 約1.55円
- 月24,000 AI応答の30%で検索した場合: 7,200 calls = 72 USD = 約11,160円/月
- 上記に加えて、検索結果を含む入力tokensがモデル単価で課金されます。

したがって、現行方針では「検索課金を発生させず、公式ソースをローカルRAG化する」方が費用・品質・監査の面で合理的です。

### 2-8. Xserverでの処理量目安

現行コードの処理は、PHP/MySQL/OpenAI API呼び出しが中心です。AI推論はOpenAI側で行うため、Xserver上でGPUは不要です。

DB増加量の目安:

- 1セッション = AI応答8回 + ユーザー8回 = 16メッセージ
- 10,000セッション/月 = 160,000メッセージ/月
- 1メッセージ平均1KBなら、本文だけで約160MB/月
- 6か月保持なら、本文約960MB + インデックス + JSONデータ
- MySQLとしては処理可能な範囲ですが、検索・一覧表示にはインデックスと保存期間管理が必要です。

平均リクエスト量の目安:

| 月間セッション | AI応答/月 | AI応答/日 | 平均AI応答/分 |
|---:|---:|---:|---:|
| 500 | 4,000 | 133 | 0.09 |
| 3,000 | 24,000 | 800 | 0.56 |
| 10,000 | 80,000 | 2,667 | 1.85 |

平均値だけならXserverでも十分処理可能です。問題になるのは、OpenAI応答待ちのPHPプロセスが同時に多数滞留するピーク時です。

AWS等への移行判断基準:

- 同時AI応答待ちが継続的に20件を超える
- P95応答時間が8秒を継続的に超える
- 月間チャットセッションが10,000件を超える
- 会話ログ保持が6か月以上で100万メッセージを超える
- KMS、RDS監査ログ、WAF詳細制御、閉域DB接続が契約要件になる
- 第三者脆弱性診断で共有サーバー設定変更の限界が指摘される

## 3. 2）セキュリティとプライバシー方針

### 3-1. 結論

本機能は、通常のFAQチャットではなく、不動産相談に伴う個人情報・機微性の高い情報を扱うシステムとして扱います。

取得し得る情報:

- 年収
- 自己資金
- 希望借入額
- 住宅ローン残債
- 他社相談状況
- 物件所在地
- 告知事項
- 氏名/電話/メール/LINE ID

そのため、以下の方針で運用します。

- OpenAI APIへ送信したデータは、モデル学習に利用されない設定/契約で運用します。
- 本番会話ログをファインチューニングや学習データとして使用しません。
- 機微情報は任意回答とし、回答拒否を可能にします。
- 連絡先保存時は同意を取得します。
- DB保存データはアプリケーションレイヤーで暗号化します。
- 会話品質レビューでは匿名化/マスキングしたログを使います。
- 保存期間と削除手順を定義します。

### 3-2. OpenAIに学習されないための契約/設定

OpenAI公式ドキュメントでは、2023年3月1日以降、OpenAI APIに送信されたデータは、明示的にオプトインしない限り、OpenAIモデルの学習・改善に使用されないとされています。

また、`/v1/chat/completions` は以下の扱いです。

- Data used for training: No
- Abuse monitoring retention: 30 days
- Application state retention: None
- Zero Data Retention eligible: Yes

本番運用での方針:

| 項目 | 方針 |
|---|---|
| 利用形態 | OpenAI API Platform |
| 学習利用 | オプトインしない |
| ファインチューニング | 本番会話ログでは行わない |
| APIキー | 本番専用ProjectのAPIキーを利用 |
| ログ保持 | 標準ではOpenAI側のabuse monitoring logは最大30日 |
| 追加統制 | 必要に応じてModified Abuse MonitoringまたはZero Data Retentionを申請 |
| DPA | 必要に応じてOpenAI Data Processing Addendumを締結 |
| 社内ルール | 本番会話ログを個人ChatGPT画面に貼り付けない |

参考:

- OpenAI Data controls: https://developers.openai.com/api/docs/guides/your-data

### 3-3. 暗号化方式

Xserver継続運用では、AWS KMS/RDS暗号化のようなインフラレイヤーの詳細制御はできません。そのため、機微情報はアプリケーションレイヤーで暗号化します。

| 対象 | 方針 |
|---|---|
| 通信 | HTTPS/TLS |
| OpenAI API通信 | HTTPS/TLS、証明書検証有効 |
| パスワード | `password_hash()` による一方向ハッシュ |
| APIキー | `secrets.php` または環境変数で管理、Git管理対象外 |
| 会話本文 | AES-256-GCMで暗号化 |
| 構造化JSON | AES-256-GCMで暗号化 |
| 連絡先 | AES-256-GCMで暗号化 |
| 会話メモリー | AES-256-GCMで暗号化 |
| バックアップ | 暗号化済みデータを保存、鍵は別管理 |

暗号化対象:

- `chat_messages.message`
- `chat_leads.structured_data`
- `chat_lead_contacts.customer_name`
- `chat_lead_contacts.phone`
- `chat_lead_contacts.email`
- `chat_lead_contacts.line_id`
- `chat_lead_contacts.raw_contact`
- `chat_session_memory.memory_json`
- `chat_session_memory.last_summary`

鍵管理:

- 暗号鍵はソースコードに置かない
- Xserver上では環境変数または非公開領域の `secrets.php` で管理
- AWS移行時はKMS/Secrets Managerへ移行
- 退職者発生、漏えい疑い、委託先変更時は即時ローテーション
- 定期ローテーションは年1回以上

### 3-4. 現行コードから見たセキュリティ上の強み

| 項目 | 確認内容 |
|---|---|
| SQL | PDO prepared statementを利用 |

## 4. 3）会話品質維持と運用体制

### 4-1. 結論

「AIが誤ったデータを学習する」という表現は、API利用の実態としてはやや正確ではありません。OpenAI APIへ送信した会話でモデルが自動的に再学習されるわけではありません。

実際のリスクは以下です。

- 古い制度情報をRAGが参照する
- RAGソース取得に失敗する
- プロンプトが曖昧で回答が一般論になる
- 顧客条件を確認せずに断定する
- 会話ログをレビューせず、改善が止まる

したがって、会話品質は「モデル任せ」ではなく、以下の4層で担保します。

1. プロンプト制御
2. 構造化ヒアリング制御
3. 公式情報RAG制御
4. 人による定期レビュー

### 4-2. 運用責任者

| 役割 | 担当 |
|---|---|
| 業務責任者 | 回答方針、営業表現、禁止表現、取得項目の承認 |
| AI品質管理者 | 会話ログレビュー、誤回答分類、改善要望 |
| シニア開発者 | プロンプト、RAG、API、DB、障害対応 |
| インフラ担当 | Xserver/AWS、cron、監視、バックアップ |
| 個人情報管理責任者 | 保存期間、削除依頼、閲覧権限、監査 |

小規模運用では兼任可能ですが、開発者だけで不動産実務上の回答品質を判断する運用は避けます。

### 4-3. 定期運用

| 頻度 | 作業 |
|---|---|
| 日次 | APIエラー、OpenAI失敗、RAG同期失敗、異常アクセス確認 |
| 週次 | 会話ログ20〜50件を匿名化レビュー |
| 週次 | 税制/補助金/金利等の高リスク回答を重点確認 |
| 週次 | RAGソース取得状況と `last_status` を確認 |
| 月次 | トークン量、API費用、チャット件数、リード化率の集計 |
| 月次 | プロンプト/RAG/ヒアリング項目の改善 |
| 四半期 | 脆弱性診断、依存関係更新、復旧テスト |
| 制度改正時 | 該当RAG更新、代表質問テスト、回答表現確認 |

### 4-4. 品質指標

| 指標 | 目標 |
|---|---:|
| AI API成功率 | 99%以上 |
| RAG同期成功率 | 95%以上 |
| 高リスク回答レビュー | 週次100%確認 |
| ランダム会話レビュー | 週20〜50件 |

### 4-5. ゴールデンテストセット

モデル変更、プロンプト変更、RAG更新時には代表質問で回帰テストします。

### 5-4. Xserverで運用する場合の技術評価

Xserverで可能な範囲:

- PHP APIの実行
- MySQLへの会話/リード/RAG保存
- cronによるRAG同期
- HTTPS運用
- 低〜中規模チャット運用

Xserverで弱い範囲:

- RDS/KMSレベルのDB暗号化証跡

- WAF詳細ルールの柔軟な管理

- SIEM/SOC連携
- 高負荷時の水平スケール
- PHPプロセス長時間占有時の細かな制御

当面の判断:

- 月間500〜3,000セッション程度: Xserverで運用可能
- 月間10,000セッション程度: 負荷試験と保存期間管理が必要
- 月間10,000超、または監査/暗号化要件が厳格化: AWS等へ移行推奨

## 6. 関連会社様への回答文面

以下は、そのままメール本文として使える文面です。

ご指摘ありがとうございます。AIチャット機能について、コスト、セキュリティ、運用体制、技術構成の4点は、正式リリース前に明文化すべき重要項目と認識しております。

まずコスト面について、本システムは会話全文や全資料を毎回AIへ無制限に送信する設計ではありません。現行コードでは、会話履歴は直近20メッセージ、AI出力は最大1,024トークン、RAG参照は最大6チャンク、RAG本文は約4,800文字で制限しています。また、購入・売却・ローン相談等のヒアリングは、まずPHP側の構造化ロジックで処理し、必要な場合のみOpenAI APIを呼び出します。

API費用は、現行モデル `gpt-4o-mini` 前提で、1AI応答あたり標準約0.19円です。月間3,000セッション、1セッション8AI応答の場合、OpenAI API費用は約4,520円/月です。月間10,000セッションでも標準試算では約15,066円/月です。モデルを上位モデルへ変更した場合は、同条件で約26,226円/月から約87,420円/月程度を見込みます。したがって、API費用よりも、会話品質レビュー、RAG更新、セキュリティ運用、障害監視の人件費が主要コストになります。

インフラについては、現行の `ai-fcard.com` はXserver上で稼働しており、サーバーは `sv16576`、Linux、Apache 2.4.x、MySQL/PHP構成です。AI推論はOpenAI側で実行されるため、GPU等は不要です。月間500〜3,000セッション程度の小〜中規模運用であれば、Xserverでも技術的には運用可能と判断しています。一方で、年収、自己資金、ローン残債、告知事項等の機微情報を本格的に扱う正式運用では、暗号化、監査ログ、アクセス制御、脆弱性診断、保存期間管理が必要です。厳格な監査やKMS/RDS/WAF等の要件が出る場合は、AWS等の専用環境へ移行する方針です。

セキュリティ面では、OpenAI API Platformを利用し、入力・出力データをモデル学習に利用しない設定で運用します。OpenAI公式ドキュメント上も、APIに送信したデータは明示的にオプトインしない限り学習に利用されません。また、`/v1/chat/completions` は学習利用なし、標準のabuse monitoring log保持は最大30日、Zero Data Retentionの対象エンドポイントです。必要に応じてModified Abuse MonitoringまたはZero Data Retentionを申請します。

保存データについては、Xserver継続運用の場合、RDS/KMSのようなインフラレイヤー暗号化を前提にできないため、会話本文、構造化JSON、連絡先、会話メモリーをアプリケーションレイヤーでAES-256-GCM暗号化します。通信はHTTPS、OpenAI API通信もTLSで行います。APIキーや暗号鍵はソースコードに置かず、環境変数または非公開設定ファイルで管理します。

会話品質については、AIを作りっぱなしにはしません。週次で会話ログ20〜50件を匿名化レビューし、税制、補助金、金利、ローン、告知事項等の高リスク回答を重点確認します。月次でプロンプト、RAGソース、ヒアリング項目を改善し、モデル変更や制度改正時には代表質問セットで回帰テストを行います。なお、OpenAI APIの利用により会話ログが自動的にモデルへ学習されるわけではありません。主なリスクは、古いRAG情報、プロンプト不備、レビュー不足による誤回答であり、これを運用で管理します。

脆弱性診断については、リリース前にSAST、Composer依存関係監査、OWASP ZAP等のDAST、手動の権限/IDOR/CORS/XSS確認を実施します。外部診断は年1回または大規模改修時に実施する方針です。現行コードでもPDO prepared statement、password_hash、ファイルアップロード検証、RAG公式ソース限定、ログインユーザーと名刺所有者の照合などの基礎対策は入っていますが、本番ではCORS制限、secure cookie、production設定、暗号化、監査ログを追加します。

開発言語はPHP、フロントエンドはHTML/CSS/JavaScript、データベースはMySQL/InnoDB/utf8mb4です。RAG情報はMySQLに保存し、OpenAIの組み込みWeb検索ツールではなく、国税庁、国土交通省、住宅金融支援機構、フラット35等の公式ソースを当社側でキャッシュして利用します。これにより、検索費用の増加を抑えつつ、回答根拠の管理と監査性を確保します。

以上の方針により、現行Xserver環境での段階的運用を可能にしつつ、機微情報保護、AI品質、将来的なAWS移行余地を含めた設計として進めます。
