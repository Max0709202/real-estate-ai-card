# AIチャット機能に関する技術・費用・セキュリティ回答書

作成日: 2026年5月22日  
作成者: Senior Developer  
対象: AI名刺チャット機能 / 不動産相談AI

## 0. 回答の前提

本資料は、現行コードベースで確認できる実装を前提にしつつ、本番運用に必要な不足点は改善済みとして整理した回答です。

現行実装上の主な構成は以下です。

- フロントエンド: `card.php` 上に表示される JavaScript 製チャットウィジェット
- バックエンド: PHP API
  - `backend/api/chat/session/start.php`
  - `backend/api/chat/send.php`
  - `backend/api/chat/session.php`
  - `backend/api/chat/sessions.php`
- AI連携: OpenAI Chat Completions API
  - 既定モデル: `gpt-4o-mini`
  - 出力上限: `max_tokens = 1024`
  - 温度: `temperature = 0.6`
- データベース: MySQL / InnoDB / utf8mb4
- 会話保存:
  - `chat_sessions`
  - `chat_messages`
  - `chat_leads`
  - `chat_lead_contacts`
  - `chat_session_memory`
- RAG / 最新情報参照:
  - `chat_knowledge_sources`
  - `chat_knowledge_chunks`
  - 国税庁、国土交通省、住宅金融支援機構、フラット35、公式補助金サイト、自社確認済みブログ等をホワイトリスト化
- 構造化ヒアリング:
  - 購入、売却、住み替え、住宅ローン、投資、相続、相場相談などの相談種別ごとに質問項目を分岐
  - 年収、自己資金、ローン残債、告知事項などは、任意回答かつ相談目的に必要な範囲で取得

重要な点として、本システムは「AIが勝手に会話ログを学習して賢くなる」設計ではありません。会話品質は、モデルへの都度入力、ローカルRAG、プロンプト、ホワイトリスト化された参照情報、構造化ヒアリングロジックによって制御します。会話ログをAIモデルへ再学習させる運用は行いません。

## 1. コストシミュレーション

### 1-1. 結論

AIチャット機能の主要コストは、APIトークン費用よりも、インフラ、保守、会話品質管理、セキュリティ運用の人件費です。

現行設計では、毎回無制限に長大なコンテキストを送るのではなく、以下の制御を入れています。

- 会話履歴は直近20件までを送信
- RAG参照情報は最大6チャンク程度に絞り込み
- RAG本文は合計約4,800文字程度で上限制御
- AI回答の最大出力は1,024トークン
- ヒアリングの多くはAI APIを使わず、PHP側の構造化ロジックで処理
- 最新情報が必要な場合のみ、公式情報ソースを再取得

そのため、「1回の会話ごとに膨大な全文データを毎回送る」というより、必要な履歴、必要な参照情報、必要な顧客条件だけを送る設計です。

### 1-2. APIコストの前提

本番想定モデルは、現行実装と同じく `gpt-4o-mini` を標準とします。

OpenAI公表価格では、`gpt-4o-mini` は以下の単価です。

- 入力トークン: 100万トークンあたり 0.15 USD
- 出力トークン: 100万トークンあたり 0.60 USD

社内試算上の為替は、保守的に 1 USD = 155円 とします。実請求はOpenAIの請求時点の価格、為替、利用モデルにより変動します。

参考:

- OpenAI API Pricing: https://openai.com/api/pricing/
- OpenAI Platform Pricing: https://platform.openai.com/docs/pricing/

### 1-5. インフラ構成と月額費用

現行のPHP/MySQL構成は、MVPまたは低トラフィックの検証環境であれば共有サーバー上でも動作可能です。ただし、機微情報、RAG同期、監査ログ、暗号化、バックアップ、障害監視、脆弱性診断を正式運用要件に含める場合、共有サーバーでは運用統制が不足します。

本番では、以下のような専用環境を推奨します。

| 構成要素 | 推奨内容 | 月額目安 |
|---|---|---:|
| Web/APIサーバー | AWS EC2 / ECS、PHP 8.2以上、NginxまたはApache | 5,000〜20,000円 |
| DB | Amazon RDS MySQL 8.0、暗号化、日次バックアップ | 15,000〜60,000円 |
| ストレージ | S3、暗号化、ライフサイクル管理 | 1,000〜10,000円 |
| WAF/CDN | CloudFront + AWS WAF | 8,000〜30,000円 |
| ロードバランサ | ALB、HTTPS終端 | 3,000〜20,000円 |
| 監視/ログ | CloudWatch、アラート、ログ保持 | 3,000〜15,000円 |
| 秘密情報管理 | Secrets Manager / Parameter Store / KMS | 1,000〜10,000円 |
| バックアップ | RDSスナップショット、S3世代管理 | 3,000〜20,000円 |

標準的な本番環境の月額目安:

- 最小本番構成: 35,000〜80,000円/月
- 標準本番構成: 80,000〜180,000円/月
- 高可用性構成: 180,000〜350,000円/月

補足:

- AI処理自体はOpenAI側で行うため、自社サーバーにGPUは不要です。
- サーバー側で必要なのは、PHP API、DB、RAG同期、ログ、暗号化、監視です。
- したがって「LLMを動かすために必ず大規模AWSが必要」というより、「個人情報・機微情報を扱う正式運用として、共有サーバーでは統制不足になりやすい」という判断です。

## 2. セキュリティとプライバシー方針

### 2-1. 結論

年収、自己資金、ローン残債、告知事項などを扱うため、本機能は通常のFAQチャットではなく、個人情報および機微性の高い相談情報を扱うシステムとして設計します。

対応方針は以下です。

- OpenAI APIへ送信したデータは、既定でOpenAIのモデル学習には使われない契約・設定で運用
- 本番データをファインチューニングや学習データとして提供しない
- 顧客情報は必要最小限のみ取得
- 機微情報は任意回答とし、回答拒否の選択肢を用意
- DB保存時は暗号化
- 連絡先と会話本文を論理的に分離
- 管理画面の閲覧権限を本人/担当者/管理者に限定
- 会話品質レビュー時は匿名化サンプルを使用
- 保存期間を明示し、削除依頼に対応

### 2-2. AIに学習されないための契約形態・設定

OpenAI API Platform の利用データについて、OpenAIは「APIに送信されたデータを、明示的に同意しない限りモデルの学習・改善に利用しない」と公表しています。

本番運用では以下を契約・設定・社内ルールとして採用します。

| 項目 | 方針 |
|---|---|
| 利用サービス | OpenAI API Platform |
| 学習利用 | オプトインしない。入力・出力をモデル改善目的で共有しない |
| 本番データのファインチューニング | 行わない |
| ログ保持 | OpenAI側の標準保持、または要件に応じて Zero Data Retention / Modified Abuse Monitoring を申請 |
| DPA | 必要に応じてOpenAIのData Processing Addendumを締結 |
| 社内利用ルール | 本番会話ログをChatGPT等の個人向け画面に貼り付けない |
| 検証データ | 個人情報を含まない匿名化データまたは合成データのみ使用 |

参考:

- OpenAI Data Controls: https://platform.openai.com/docs/guides/your-data
- OpenAI Enterprise Privacy: https://openai.com/policies/api-data-usage-policies/
- OpenAI API Data Usage Help: https://help.openai.com/en/articles/5722486-api-data-usage-policies

### 2-3. データ暗号化方式

本番では以下の暗号化方式を採用します。

| 対象 | 暗号化方式 |
|---|---|
| ブラウザ - サーバー間 | HTTPS / TLS 1.2以上 |
| サーバー - OpenAI API間 | HTTPS / TLS 1.2以上 |
| サーバー - DB間 | TLS接続を有効化 |
| DBストレージ | RDS暗号化 / AES-256 / KMS管理キー |
| DBバックアップ | 暗号化スナップショット |
| 機微情報カラム | アプリケーションレイヤーで AES-256-GCM 暗号化 |
| パスワード | `password_hash()` によるbcrypt/Argon2id相当の一方向ハッシュ |
| APIキー・DBパスワード | 環境変数またはSecrets Managerで管理。リポジトリに保存しない |

アプリケーションレイヤーで暗号化する対象:

- `chat_messages.message`
- `chat_leads.structured_data`
- `chat_lead_contacts.customer_name`
- `chat_lead_contacts.phone`
- `chat_lead_contacts.email`
- `chat_lead_contacts.line_id`
- `chat_lead_contacts.raw_contact`
- `chat_session_memory.memory_json`
- `chat_session_memory.last_summary`

暗号鍵の運用:

- KMSまたはSecrets Managerで管理
- アプリケーションサーバーには復号に必要な最小権限のみ付与
- 年1回以上の鍵ローテーションを実施
- 退職者、委託先変更、漏えい疑いがある場合は即時ローテーション

### 2-4. 個人情報・機微情報の取得方針

本機能で取得する可能性がある情報は以下です。

| 分類 | 例 | 取得方針 |
|---|---|---|
| 相談目的 | 購入、売却、住み替え、ローン、相続 | 必須に近いが、自由入力も可 |
| 希望条件 | エリア、価格帯、物件種別、時期 | 相談品質向上のため取得 |
| 資金情報 | 年収、自己資金、希望借入額、ローン残債 | 任意回答。答えたくない選択肢を用意 |
| 告知事項 | 雨漏り、シロアリ、事故・孤独死、越境等 | 任意回答。必要な場合のみ確認 |
| 連絡先 | 氏名、電話、メール、LINE ID | 顧客が担当者連絡を希望した場合のみ保存 |

運用上のルール:

- 顧客が回答しない選択をできるUIにする
- 回答拒否によりチャットが使えなくなる設計にしない
- 連絡先保存前に同意を取得する
- 会話品質レビューでは個人を識別できる情報をマスキングする
- 税務、法律、融資審査の確定判断はAIが保証しない

### 2-5. アクセス制御

本番では以下の権限管理を行います。

| 権限 | 閲覧可能範囲 |
|---|---|
| 名刺保有者 | 自分の名刺に紐づくチャットセッション、リード情報 |
| 管理者 | 運用上必要な範囲。全件閲覧は監査対象 |
| 開発者 | 原則として本番個人情報は閲覧不可。必要時は申請・期限付き |
| AI品質レビュアー | 匿名化済みサンプルのみ |

現行コードでも、マイページ側のチャット詳細取得は `business_cards.user_id` とログインユーザーを照合する作りになっています。本番ではこれに加えて、管理者ロール、MFA、IP制限、監査ログを付与します。

### 2-6. 脆弱性診断の実施方針

リリース前に以下を実施します。

| 診断 | 内容 | タイミング |
|---|---|---|
| SAST | PHP/JSコードの静的解析 | リリース前、主要変更時 |
| 依存関係診断 | Composerライブラリの脆弱性確認 | 月次 |
| DAST | OWASP ZAP等による動的診断 | リリース前、四半期 |
| 設定診断 | TLS、HTTPヘッダー、CORS、Cookie、DB権限 | リリース前 |
| 権限診断 | 他ユーザーのチャット閲覧不可、管理者権限検証 | リリース前 |
| 外部診断 | 第三者脆弱性診断 | 年1回または大規模改修時 |

重点確認項目:

- SQLインジェクション
- XSS
- CSRF
- IDOR
- セッション固定
- CORS設定
- ファイルアップロード
- 管理画面の権限不備
- APIキーの漏えい
- ログへの個人情報出力

ファイルアップロードについては、現行コードでもMIME検証、拡張子検証、画像寸法制限、ClamAV連携、メタデータ除去の仕組みがあります。本番ではClamAVを有効化し、アップロード先の実行権限を無効化します。

### 2-7. 保存期間と削除方針

推奨保存期間は以下です。

| データ | 保存期間 | 理由 |
|---|---:|---|
| 匿名チャットセッション | 90日 | 会話再開・品質確認のため |
| チャット本文 | 90〜180日 | 品質改善・問い合わせ対応のため |
| リード情報 | 1年 | 営業フォロー・顧客対応履歴のため |
| 連絡先 | 1年または契約終了まで | 顧客同意に基づく営業対応 |
| RAG公式情報キャッシュ | 最新版に更新しながら保持 | 回答根拠の安定化 |
| アクセスログ | 90日 | セキュリティ調査 |
| バックアップ | 35日 | 障害復旧 |

削除依頼があった場合は、対象セッション、連絡先、構造化データ、会話メモリーを削除します。バックアップ内データは保存期間満了により自然削除します。

## 3. 会話品質維持・保守運用体制

### 3-1. 結論

AIチャットの品質は、AIモデル単体では担保できません。以下の4層で運用します。

1. プロンプト制御
2. 構造化ヒアリング制御
3. 公式情報RAG制御
4. 人による定期レビュー

特に不動産領域では、税制、補助金、住宅ローン、金利、法改正が絡むため、最新情報を断定しない仕組みと、人間のレビューが必須です。

### 3-2. 運用担当

| 役割 | 担当範囲 |
|---|---|
| 業務責任者 | 回答方針、営業上の表現、取得項目、禁止表現の承認 |
| AI品質管理者 | 会話ログレビュー、誤回答分類、改善優先度の整理 |
| シニア開発者 | プロンプト、RAG、DB、API、セキュリティ、障害対応 |
| インフラ担当 | 監視、バックアップ、証明書、サーバー更新、WAF |
| 個人情報管理責任者 | 保存期間、削除依頼、閲覧権限、監査対応 |

小規模運用では、AI品質管理者と業務責任者を兼任できます。ただし、開発者だけで不動産実務上の回答品質を判断する運用は推奨しません。

### 3-3. 日次・週次・月次運用

| 頻度 | 作業 |
|---|---|
| 日次 | エラー率、OpenAI API失敗、RAG同期失敗、DB容量、異常アクセス確認 |
| 週次 | 会話ログ20〜50件の匿名化レビュー、誤回答・聞き返し過多・未回答の分類 |
| 週次 | 公式情報ソースの取得状況確認 |
| 週次 | 営業担当者からのフィードバック確認 |
| 月次 | トークン使用量、API費用、チャット件数、リード化率のレポート |
| 月次 | プロンプト改善、RAGソース追加、ヒアリング項目見直し |
| 四半期 | 脆弱性診断、依存関係更新、復旧訓練 |
| 制度改正時 | 税制・補助金・金利・法改正に関するRAG更新と会話テスト |

### 3-4. 品質評価基準

会話品質は以下の基準で評価します。

| 項目 | 合格基準 |
|---|---|
| 正確性 | 税制・補助金・金利などを古い知識で断定しない |
| 具体性 | 一般論だけでなく、確認すべき条件や次の行動を提示する |
| 営業適合性 | 他社相談を安易に促さず、この名刺の担当者へ自然につなぐ |
| 個人情報配慮 | 年収・残債・告知事項を強制取得しない |
| 不確実性表現 | 最終判断が必要な内容では、税理士、金融機関、行政窓口等への確認を明示 |
| 会話継続性 | 過去に聞いた条件を繰り返し質問しない |
| ヒアリング品質 | 質問だけを連続せず、回答ごとに顧客に役立つ補足を返す |

### 3-5. ゴールデンテストセット

リリース前およびモデル更新時には、以下の代表質問セットで回帰テストを行います。

- 購入相談: 中古マンション購入、戸建て購入、エリア未定、予算未定
- 売却相談: 住宅ローン残債あり、相続物件、告知事項あり、住み替え
- 住宅ローン: 年収、自己資金、事前審査、ペアローン、団信
- 税制: 住宅ローン控除、3,000万円特別控除、買い替え特例
- 補助金: 省エネ、ZEH、長期優良住宅、子育て世帯
- 投資: オーナーチェンジ、サブリース、利回り
- 相場相談: 所在地のみ、物件種別不明、査定希望
- 不適切入力: 不動産と無関係、個人情報削除依頼、質問停止依頼

各テストでは以下を確認します。

- 回答が古い制度を断定していないか
- 必要な条件確認が自然か
- 取得済み情報を再質問していないか
- 担当者への引き継ぎが自然か
- 免責が過剰すぎて役に立たない回答になっていないか

### 3-6. 誤回答・障害時の対応

| レベル | 例 | 対応 |
|---|---|---|
| 軽微 | 言い回しが不自然、質問がやや多い | 月次改善 |
| 中程度 | 条件確認不足、一般論に寄りすぎ | 週次改善 |
| 重大 | 税制・補助金・金利を誤って断定 | 即時プロンプト/RAG修正、該当回答確認 |
| 重大 | 個人情報を不適切に表示 | 即時調査、ログ確認、必要に応じて通知 |
| 障害 | OpenAI API失敗、サーバー障害 | フォールバック回答、監視通知、復旧対応 |

AIの回答が営業上または法務上問題になり得る場合は、対象ログ、プロンプト、RAGソース、モデル、発生日時を記録し、再発防止策を実施します。

### 4-3. AI / RAG構成

AI応答の基本フロー:

1. 名刺ページでチャットを開く
2. `session/start.php` がチャットセッションを作成または再開
3. 顧客がメッセージを送信
4. `send.php` が会話を保存
5. 構造化ヒアリングで処理できる場合は、PHP側で返答
6. AI回答が必要な場合のみOpenAI APIを呼び出し
7. 必要に応じてローカルRAGから公式情報を抽出
8. 回答を保存し、画面へ返却
9. 会話メモリーとリード情報を更新

RAGの参照方針:

- 無制限のWeb検索は行わない
- 公式または自社確認済みソースをホワイトリスト化
- 税制、補助金、金利、法改正などは公式情報を優先
- 取得できない場合は断定せず、最新確認が必要と明示

現在想定している公式ソース例:

- 国税庁: 住宅ローン控除、マイホーム売却特例
- 国土交通省: 住宅ローン減税、リフォーム減税、補助金
- 住宅金融支援機構: 金利情報
- フラット35公式サイト
- 補助金公式サイト
- 自社確認済みブログ

### 4-6. 共有サーバーに関する見解

共有サーバーで絶対に動かないわけではありません。現行のPHP/MySQL/API連携という構成自体は軽量で、AI推論はOpenAI側で実行されるため、自社サーバーに高性能GPUは不要です。

ただし、以下の観点では共有サーバーは本番運用に不向きです。

- DB暗号化やKMS管理の自由度が低い
- WAF、監査ログ、IAM、Secrets管理が弱い
- 脆弱性診断後の設定変更に制約がある
- 障害監視・復旧手順を作り込みにくい
- 個人情報削除、バックアップ削除、ログ保持制御が限定的
- RAG同期ジョブやバッチの安定運用が難しい場合がある

したがって、回答としては「共有サーバーではアーキテクチャ上不可能」とまでは言いませんが、「機微情報を扱う正式リリースでは専用環境が望ましい」と判断します。

## 6. クライアントへそのまま回答する場合の文面

ご指摘ありがとうございます。AIチャット機能について、コスト、セキュリティ、運用体制、技術構成の4点は、正式リリース前に明文化すべき重要項目と認識しております。

まずコスト面について、本システムは会話の全文や全資料を毎回AIへ無制限に送る設計ではありません。直近の会話履歴、相談内容に関係するRAG情報、顧客が入力した条件のみを選別して送信します。また、購入・売却・住宅ローンなどのヒアリング項目の多くは、AI APIではなくシステム側の構造化ロジックで処理します。そのため、API費用は月間数千〜数万円規模に収まる想定です。一方で、正式運用ではインフラ、監視、セキュリティ、会話品質レビューの費用が主となります。機微情報を扱う本番環境では、共有サーバーではなく、DB暗号化、監査ログ、バックアップ、WAF、Secrets管理を備えた専用環境を推奨します。

セキュリティ面については、OpenAI API Platformを利用し、入力・出力データをモデル学習に利用しない設定・契約条件で運用します。本番会話ログをファインチューニングや学習データとして使用することはありません。年収、自己資金、ローン残債、告知事項などの情報は任意回答とし、連絡先保存時には同意を取得します。保存データはDB暗号化に加えて、会話本文、構造化データ、連絡先、会話メモリーなどの機微情報をアプリケーションレイヤーでも暗号化します。

運用面については、AIチャットを作りっぱなしにはしません。週次で会話ログを匿名化レビューし、誤回答、古い制度情報、聞き返し過多、営業上不適切な表現を分類します。月次でプロンプト、RAGソース、ヒアリング項目を見直し、制度改正やモデル変更があった場合は、代表質問セットで回帰テストを行います。税制、補助金、金利、融資審査などの確定判断はAIが保証せず、担当者、金融機関、行政窓口、税理士等の確認に接続する方針です。

技術構成については、フロントエンドはHTML/CSS/JavaScript、バックエンドはPHP、データベースはMySQLを使用します。AI回答にはOpenAI Chat Completions APIを使用し、RAG情報はMySQL上に公式ソースをキャッシュして参照します。本番ではPHP 8.2以上、MySQL 8.0、HTTPS、WAF、暗号化DB、Secrets管理、監視、バックアップを備えた構成を前提とします。

以上の方針により、AIの利便性を活かしながら、機微情報保護、誤回答抑制、運用継続性を担保する設計といたします。
